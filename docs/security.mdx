Arktion holds custody of an admin keypair that mints tokens and sponsors every transaction, and it issues invisible wallets to users who have never touched crypto. That makes the trust boundary the most important part of the system. This page describes how it is enforced.

## zkLogin Authentication

Users sign in with Google and receive a Sui wallet — no seed phrase, no extension.

<Steps>
  <Step title="Verify the Google ID token">
    The backend validates the Google JWT against Google's published JWKS keys, pinned to the `RS256` algorithm, with both the audience (our Google client ID) and the issuer (`accounts.google.com`) checked. Required claims (`sub`, `iss`, `aud`) must be present. A forged or mis-issued token is rejected before anything else happens.
  </Step>
  <Step title="Derive the wallet">
    The salt is managed by Enoki and the resulting Sui address is derived deterministically from the verified JWT. The same address is used when the user row is created, so it always matches the address a ZK proof is generated for.
  </Step>
  <Step title="Issue a session">
    Only after JWT verification succeeds does the backend create a session and return its own session token.
  </Step>
</Steps>

## Stateful, Revocable Sessions

The session token is a JWT, but it is **not** trusted on its signature alone. Every authenticated request runs two checks:

1. Verify the JWT signature against `JWT_SECRET` (catches forgery).
2. Look up the session row in Postgres (catches revocation).

The second check is what makes logout real: deleting or revoking the session row invalidates the token **immediately**, even though its `exp` claim hasn't expired. The guard also rejects sessions that are expired or belong to a deactivated account. This applies to both the user guard and the admin guard.

<Info>
  A stolen token is useless the moment the user logs out or an admin deactivates the account — there is no "valid until it expires" window.
</Info>

## Admin Security

Admin accounts can mint tokens and manage users, so they are held to a higher standard than reader accounts.

- **Passwords** are hashed with bcrypt (cost 12). Repeated failed logins lock the account temporarily.
- **TOTP is mandatory.** The TOTP secret is encrypted at rest with AES-256-GCM (not just stored), and one-time backup codes are individually bcrypt-hashed.
- **Sessions use refresh-token rotation** — each refresh issues a new token and revokes the old one; refresh tokens are stored only as SHA-256 hashes.
- **Ranked roles** — Super Admin, Moderator, Reviewer — enforced by a role guard with a strict rank check, so no admin can act on a peer of equal or higher rank.
- **Audit logging** — every privileged action is written to an append-only audit log with the acting admin, target, IP, and route.

### Four-Eyes Role Changes

Assigning or changing an admin role cannot be done by one person. It is a two-step flow:

1. A Super Admin **requests** the change (recorded as `PENDING`).
2. A **different** Super Admin must **approve** it before it takes effect. The requester can never approve their own request, and neither party may act on their own account.

On approval the target's sessions are revoked so the new role takes effect immediately rather than waiting for the old token to expire.

## Gasless, Sponsored Transactions

Users never hold SUI. For transactions that need gas, the backend sponsors them:

- The user signs the transaction with their zkLogin key (their wallet is the sender).
- The gas-sponsor keypair co-signs as gas payer and submits.
- Before sponsoring, the service checks the gas treasury balance and **refuses new transactions** if it's below threshold — failing fast with a clear error instead of letting individual transactions die with cryptic gas faults.

USDC tips and transfers are free on Sui, so those are signed by the user and submitted directly — no sponsor co-signature needed.

## Transaction Integrity & Idempotency

Because the backend submits on users' behalf and networks retry, every value-moving action is idempotent.

- **INK minting** uses a deterministic key (`ink:{userId}:{trigger}:{seriesId}:{context}`) checked in Postgres first; the `ink_earning` Move contract **also** enforces it on-chain, so a milestone can never be double-minted even if the Postgres mirror drifts.
- **Tips and sends** carry a client idempotency key with a unique DB constraint; a retried request returns the existing pending record instead of creating a duplicate, and the on-chain layer rejects a replayed signed transaction.
- **Reading progress is monotonic** — it only ever moves forward, so an automatic re-read can't regress a reader's history or re-fire earned milestones.
- **On-chain passport stats are admin-attested.** The passport is owned by the reader, so they sign the update — but the values are signed off-chain by the admin key and verified on-chain (Ed25519) against a stored public key, with a monotonic guard against replay. A reader can submit and pay nothing, but cannot forge a higher level or fake INK.

## Input Handling

- **Frontend** validates user input with Zod before submitting, and never trusts URL params or local storage without sanitisation. No tokens or wallet addresses are placed in URL query strings.
- **Backend** validates every DTO with `class-validator` under a global `ValidationPipe({ whitelist: true, forbidNonWhitelisted: true })`, so unexpected fields are stripped or rejected. All database access goes through Prisma (parameterised — no string-interpolated queries). `helmet` sets security headers and CORS is locked to the configured origin in production.
- **Fail-fast config** — required environment variables are validated at boot; the app refuses to start on a missing or malformed secret rather than failing later at runtime.
